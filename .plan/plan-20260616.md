## Phase R — 审查修复（v0.50.3）

### 可靠性 / 功能正确

#### [x] R01：scheduler — upsertRepeatableJob remove 失败不再阻断 queue.add

- **问题**：`removeRepeatableByKey()` 失败时整个 upsert 回滚，`queue.add()` 永远不执行，用户无 sync 任务。
- **修复**：在 `upsertRepeatableJob` 内部拆分 try/catch，remove 步骤失败只 warn，add 始终执行。
- **影响文件**：`apps/api/src/jobs/scheduler.ts`

#### [x] R02：push — sendNotification 缺少 TTL，离线设备永远收不到提醒

- **问题**：`webpush.sendNotification()` 无第三个 options 参数，默认 TTL=0，Push 服务单次投递失败即丢弃。
- **修复**：添加 `{ TTL: 86400 }`，消息在 Push 服务队列中保留 24 小时。
- **影响文件**：`apps/api/src/lib/push.ts`

#### [x] R03：SettingsPage — disablePush 后端失败时 setPushEnabled(false) 被跳过

- **问题**：`disablePush()` 在浏览器取消订阅后调用后端 API，若后端抛错，`setPushEnabled(false)` 被跳过，UI 显示"停用"但浏览器订阅已销毁。
- **修复**：catch 块中重新从浏览器同步实际订阅状态（`getExistingSubscription().then(sub => setPushEnabled(!!sub))`）。
- **影响文件**：`apps/web/src/pages/SettingsPage.tsx`

#### [x] R06：scheduler — airing-reminders 失败被标记为 "completed" 而非 "failed"

- **问题**：catch 块只 log 不 re-throw，BullMQ 认为任务成功，不触发重试。
- **修复**：catch 末尾添加 `throw e`。
- **影响文件**：`apps/api/src/jobs/scheduler.ts`

### 安全性

#### [x] R04：CSV sanitizer — leading-space 绕过 + 循环 tab 防御

- **问题一**：值 `" =IMPORTXML(...)"` 首字符为空格，regex `^[=+\-@\t]` 不匹配，公式被原样导出，Google Sheets 可能执行。
- **问题二**：`\t` 同时出现在触发器集合和前缀字符中，防御逻辑循环矛盾。
- **修复**：改用 `clean.trimStart()` 检测（暴露空格后的公式字符），前缀改为空格（非触发字符），移除 `\t` 出触发器集合。
- **影响文件**：`apps/api/src/routes/history.ts`、`apps/api/src/__tests__/history-routes.test.ts`

#### [x] R07：notifications — 无订阅数量上限，可 DoS 推送 worker

- **问题**：认证用户可无限 POST `/notifications/subscribe`，推送 worker 会对单个用户并行发送数千条推送。
- **修复**：subscribe 路由插入前先 count，≥10 条返回 429。
- **影响文件**：`apps/api/src/routes/notifications.ts`

### 可靠性

#### [x] R05：push — VAPID_SUBJECT 轮换不被感知

- **问题**：`ensureVapid()` 只比较 publicKey/privateKey，subject-only 热轮换无效。
- **修复**：新增 `configuredSubject` 模块变量，纳入变更检测条件。
- **影响文件**：`apps/api/src/lib/push.ts`

#### [x] R08：sendPush — 不安全的 statusCode 类型断言导致 410 漏判

- **问题**：`(e as { statusCode?: number }).statusCode` 对 plain Error 返回 undefined，死订阅永不被剪枝。
- **修复**：改用 `instanceof` + `"statusCode" in e` 双重 guard。
- **影响文件**：`apps/api/src/lib/push.ts`

#### [x] R09：SettingsPage — Promise.all 吞掉错误导致 push 状态不一致

- **问题**：`Promise.all([...]).catch(()=>{})` 任意一侧失败会让另一侧已设置的状态残留，按钮显示错误。
- **修复**：两个 Promise 改为独立链（`.then(...).catch(()=>{})`），互不干扰。
- **影响文件**：`apps/web/src/pages/SettingsPage.tsx`

#### [x] R10：sw.js — skipWaiting() 在 waitUntil 外调用，cache 失败时 SW 仍激活

- **问题**：`skipWaiting()` 在 `cache.add()` 之前同步调用，网络离线时缓存写入失败但 SW 仍激活，offline shell 为空。
- **修复**：将 `skipWaiting()` 链入 `waitUntil` 的 `.then()` 末尾。
- **影响文件**：`apps/web/public/sw.js`

### UX / 正确性

#### [x] R11：enablePush — 重复点击创建孤立订阅端点

- **问题**：快速双击"启用"时两次 `pushManager.subscribe()` 并发，browser 可能返回不同 endpoint，两条都入库，每天收到双倍推送。
- **修复**：`subscribe()` 前先调 `reg.pushManager.getSubscription()`，已有订阅则复用。
- **影响文件**：`apps/web/src/lib/push.ts`

#### [x] R12：airing-reminders — "+N" 计数的是 episode 数而非 show 数

- **问题**：`airing.length - 1` 在剧集批量发布时暗示"+3 个其他剧集"，实际上可能是同一部剧的多集。
- **修复**：用 `new Set(airing.map(e => e.showTitle)).size - 1` 计算 distinct show 数。
- **影响文件**：`apps/api/src/jobs/airing-reminders.ts`

### 测试覆盖

#### [x] R13：notifications 测试 — 缺少 VAPID_SUBJECT 缺失的半配置场景 + 429 上限场景

- **新增测试**：SUBJECT 缺失时 GET /vapid-public-key 返回 200 但 POST /subscribe 返回 503；用户已有 10 条订阅时返回 429。
- **更新 mock**：`createMockDb` 支持 `count()` 查询返回。
- **影响文件**：`apps/api/src/__tests__/notifications-routes.test.ts`
