## 执行原则

- 与 api 侧不同，本计划**不追求一次性覆盖全部文件**——体量分散在 35+
  个文件，逐一"读全文→写测试→验证→提交"的节奏预计需要跨越多个会话；
  本文件按层级/主题分组批次，后续任何一次执行都可以从某个批次继续，
  不需要重新做结构分析。
- 复用已验证的测试模式：
    - 页面组件：`render` + `QueryClientProvider`（`retry:false`）+
      `MemoryRouter`，mock `../../hooks` 与/或 `../../lib/api`（参照
      `HistoryPage.test.tsx`/`ShowDetailPage.test.tsx`）。
    - hooks：`renderHook` + `QueryClientProvider` wrapper，mock
      `../../lib/api`（参照 `notes-lists-ratings.test.tsx`）。
    - `lib/api.ts` 本身：`vi.stubGlobal("fetch", ...)`，不 mock
      `../../lib/api`——直接验证 URL/method/body 构造与错误处理（参照
      api 侧 `trakt-client.test.ts` 的表驱动写法）。
- 纯展示型组件（无数据请求、无副作用）优先用最简单的 `render` +
  `screen` 断言，不需要 QueryClientProvider/mock。
- 每个子批次：`tsc --noEmit`（web 包）+ 全套件测试全绿（只增不减）+
  `vitest run --coverage` 确认涨幅 + 独立 commit；不阈值造假——ratchet
  设在实测值以下。
- 大型/超大型文件（HeroSection/JellyfinTab/MovieDetailPage/
  ListsPage/CollectionPage/BackupTab）执行前必须先通读全文，具体测试
  分支在通读后于本文件对应批次补充细化（沿用 api 侧
  plan-20260715.md 对 sync.ts 的方法论）。

### 批次 3 — `src/lib/push.ts`（137 行，1.8%）✅

- [x] `isPushSupported`/`getExistingSubscription`/`fetchVapidPublicKey`：
      支持性检测（`serviceWorker`/`PushManager`/`Notification` 三者
      都存在才算支持）、降级返回 null（fetch 失败/不支持）。
- [x] `enablePush` 全部分支：无 VAPID key → `server-unconfigured`；
      用户拒绝授权 → `permission-denied`；无现有订阅 → 全新 subscribe；
      现有订阅 key 匹配 → 复用（不重新 subscribe）；key 已轮换 →
      unsubscribe 旧订阅后重新 subscribe；unsubscribe 失败但浏览器已
      清理 → 照常走全新 subscribe；unsubscribe 失败但重新检查发现
      **并发标签页已经用新 key 订阅成功** → 直接注册该订阅到后端并提前
      return（不再调用 `pushManager.subscribe`）；unsubscribe 失败且
      重新检查发现旧 key 仍在生效 → 抛 `push-rotation-blocked`；
      unsubscribe 失败且重新检查 `getSubscription()` 本身也失败
      （SW 注册退化）→ 视为已清理，照常走全新 subscribe。
- [x] `disablePush`：无现有订阅 → 直接返回不调用任何 API；有现有订阅 →
      先浏览器 `unsubscribe()` 再通知后端。
- [x] mock 浏览器 API 用 `vi.stubGlobal("Notification"/"PushManager", ...)` + `Object.defineProperty(navigator, "serviceWorker", ...)`；VAPID
      key 用两个可精确字节比对的 base64url 测试串（`"AAAA"`→`[0,0,0]`、
      `"AAAB"`→`[0,0,1]`）避免逐字节手算 base64 解码。
