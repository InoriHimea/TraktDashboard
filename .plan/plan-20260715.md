## 执行原则

- 沿用已验证的 mock 模式：路由测试整体 mock 依赖的 service 模块
  （`../services/sync.js`、`../services/trakt.js`、`../services/jellyfin.js`、
  `../services/tmdb.js`）+ DB chain-mock；service 测试 mock 更底层的
  `providerFetch`（`../lib/http.js`，已被 `p1-http.test.ts` 覆盖，本轮
  只关心业务分支不重复测 HTTP 层）。
- `services/trakt.ts` 的 `traktFetch`/`traktFetchRaw` 是 `getTraktClient()`
  内部闭包，无法单独 mock/导出——测试通过 mock `providerFetch` 的返回值
  间接驱动，鉴权/token 部分用一个"始终有效未过期"的 mock token 短路掉，
  避免重复测已经被 `trakt-token-refresh.test.ts` 覆盖的刷新逻辑。
- `routes/shows.ts` 的 `POST .../watch` 内的 fire-and-forget IIFE（不
  `await`，错误被空 catch 吞掉）：测试策略是——响应本身必须在 IIFE 完成
  前就返回 200（验证"never blocks the response"）；IIFE 内部分支（是否
  调用 `autoDeleteJellyfinEpisode`）通过请求后 `await` 一次
  `Promise.resolve()`/`setImmediate` 让微任务队列 flush 后再断言 mock
  调用情况来验证四个门槛条件（jellyfin 配置存在 + autoDeleteLibraryIds
  非空 + show 状态 ended/canceled + 进度 100% completed）分别生效。
- `services/sync.ts` 每个批次开始前，先针对该批次涉及的具体函数做一次
  完整源码通读（本计划目前只掌握函数边界/行号，尚未逐行读取内部实现），
  沿用此前"先读全文再写测试"的方法论；不在本计划阶段预先读完全部
  2328 行。
- 每个子批次独立验证（tsc + 全套件 + 覆盖率）+ 独立 commit；不触碰真实
  Trakt/Jellyfin/TMDB 服务或生产数据库。
- 若本计划执行到 `services/sync.ts` 部分（批次 6-10）跨会话延续，允许
  拆分成独立的后续 plan 文件（如 plan-20260716.md 起）承接，只要
  requirement.md 与本文件的批次勾选状态保持同步。
