# Plan 2026-07-13 — N3-T02 覆盖率回升（第三批：backup + jellyfin）

## 背景

N3-T02 前两批（2026-07-12）把 api 覆盖率从 22.4% 拉到 41.0%（stmts），补齐了
collection/lists/stats/ratings/notes/auth-middleware/system/trakt/discover/search/
img/auth 共 12 个路由文件，四个文件（trakt/discover/img/auth）达到 100%。

剩余零覆盖的两个文件是全仓库 **blast radius 最大**的代码：

- `services/backup.ts`（953 行，3.72%）+ `routes/backup.ts`（980 行，5.02%）——
  `restoreDatabase()` 会 `DROP SCHEMA public/drizzle CASCADE` 后重放 pg_dump，
  一旦回归漏检，恢复接口可能把生产库炸穿；四个云 provider（GDrive/OneDrive/
  WebDAV/S3）的上传/下载/清理全部无测试。
- `services/jellyfin.ts`（494 行，0%）+ `routes/jellyfin.ts`（570 行，0%）——
  `deleteJellyfinItem` / `autoDeleteJellyfinEpisode` 是全仓库唯一真删 Jellyfin
  媒体文件的调用点（job 层 `jellyfin-auto-delete.ts` 已有单测，但它依赖的这两个
  service 函数本身从未被直接测试过）。

这两块体量合计约 3000 行，是目前 api 端最大的单块缺口，价值明显高于继续测
web 页面组件。

## 执行原则

- **绝不联真实服务**：所有测试必须 mock `fetch`（Jellyfin/云 provider HTTP 调用）
  和 `child_process.spawn`（pg_dump/psql），任何情况下不允许在测试中触发真实
  网络请求或真实 shell 命令。
- **测试只读，不代表可以跑真实自动删除**：这批工作是给 `deleteJellyfinItem`
  等函数补单测（mock 调用），不涉及、也不会执行 `runJellyfinAutoDelete()`
  或任何针对生产 Jellyfin/数据库的真实操作（沿用项目既有纪律）。
- 复用已验证的 mock 模式：HTTP provider 用 `vi.stubGlobal("fetch", ...)`
  （参考 `img-routes.test.ts`/`auth-routes.test.ts`）；DB chain-mock 沿用
  `collection-routes.test.ts` 的 SelectBuilder/ChainBuilder 风格。
- 每个子批次独立验证（tsc + 全套件 + 覆盖率）+ 独立 commit，避免一次改动过大
  难以定位问题。
